# SWCC Design Document

**Socialism With Chinese Characteristics — SW Claude Code**

民主集中制多智能体编排 Claude Code 插件

## 核心理念

将中国特色社会主义政治体制的"民主集中制"引入多 Agent 协作：
- **民主阶段**：政协左右派协商，充分暴露分歧
- **集中阶段**：党委综合裁决，一锤定音
- **执行阶段**：国务院拆分任务，部委并行执行
- **监察阶段**：纪委 code review + 自动化验证

与 directive（美式三权分立）和 edict（中国帝制三省六部）的区别：
- 不搞互相否决（避免美式扯皮）
- 不搞一人独裁（避免帝制盲区）
- 先充分讨论，再集中决策，执行力极强

## 命令定义

| 命令 | 用法 | 说明 |
|------|------|------|
| `/zhili` | `/zhili [--scale S] 任务描述` | 全流程：中办→政协→党委→国务院→部委→纪委 |
| `/xieshang` | `/xieshang 任务描述` | 仅政协协商，产出多方案报告，不执行 |
| `/zhixing` | `/zhixing 方案描述` | 跳过协商，直接走国务院→部委→纪委 |
| `/jicha` | `/jicha` | 对当前代码变更做纪委监察（review+测试） |
| `/juguo` | `/juguo 任务描述` | 举国体制：跳过协商，全 agent 并行爆发，纪委仅跑自动化 |

## Agent 定义

### zhongban（中央办公厅）

- **角色**: 收文分拣，判定任务规模
- **模型**: opus
- **工具**: Glob, Grep, Read
- **输入**: 用户任务描述
- **输出**: `{ scale: "小|中|大", context: "代码库相关摘要", reasoning: "判定理由" }`
- **判定标准**:
  - 小：涉及文件 <3 且逻辑简单（bug fix、小改动）
  - 中：涉及 3-8 文件，有一定复杂度
  - 大：>8 文件 或 架构级变更

### zuopai（左派委员）

- **角色**: 政协左派，主张大胆革新
- **模型**: opus
- **工具**: WebSearch, WebFetch, Grep, Glob, Read
- **核心 prompt**: "你是政协左派委员。你主张大胆革新——引入新技术、重构旧代码、不惜短期阵痛换长期收益。研究业界最新实践(SOTA)，如果现有方案陈旧落后，你要敢于推倒重来。给出具体的代码 diff 草案。"
- **输出**: 创新方案报告（含代码 diff 草案）

### youpai（右派委员）

- **角色**: 政协右派，主张稳中求进
- **模型**: opus
- **工具**: Grep, Glob, Read, Bash
- **核心 prompt**: "你是政协右派委员。你主张稳中求进——优先复用现有代码、保持向后兼容、最小化改动范围。能修补的绝不重写，能复用的绝不新建。验证激进方案的可行性风险。给出具体的代码 diff 草案。"
- **输出**: 稳健方案报告（含代码 diff 草案）

### zhongjian（中间路线）

- **角色**: 实事求是，调和左右
- **模型**: opus
- **工具**: Grep, Glob, Read
- **触发条件**: 仅大任务时参与
- **核心 prompt**: "你是政协中间路线委员。你实事求是——分析左右两派方案的优劣，取其精华去其糟粕，提出一个平衡创新与稳定的折中方案。给出具体的代码 diff 草案。"
- **输入**: 左派报告 + 右派报告
- **输出**: 折中方案报告（含代码 diff 草案）

### dangwei（党委）

- **角色**: 集中决策，最终裁决
- **模型**: opus
- **工具**: Read
- **核心 prompt**: "你是党委。你的职责是集中决策——综合政协各派意见，批判性地审视每个方案。对左倾冒险主义要批评，对右倾保守主义要纠正。产出一份明确的、可执行的最终实施方案，包含具体的文件变更清单和实施顺序。"
- **输入**: 全部委员报告
- **输出**: 结构化实施方案（文件清单、变更描述、实施顺序、验收标准）

### guowuyuan（国务院）

- **角色**: 执行规划，拆分子任务
- **模型**: opus
- **工具**: Grep, Glob, Read
- **核心 prompt**: "你是国务院。你的职责是将党委的实施方案拆解为可并行执行的具体子任务。每个子任务必须明确：负责的文件、具体变更内容、与其他子任务的依赖关系。无依赖的任务应标记为可并行。"
- **输入**: 党委方案
- **输出**: `[{id, files, description, depends_on}, ...]`

### buwei（部委 / 执行 agent）

- **角色**: 执行具体代码变更
- **模型**: opus
- **工具**: Read, Edit, Write, Bash, Grep, Glob
- **实例化**: 动态生成，每个子任务一个实例，按依赖关系分批并行
- **输入**: 单个子任务描述 + 相关文件
- **输出**: 代码变更

### jiwei（中央纪委）

- **角色**: 监察验收
- **模型**: opus
- **工具**: Read, Grep, Glob, Bash
- **核心 prompt**: "你是中央纪委。你的职责是监察验收——对所有代码变更进行严格审查。第一步：代码审查（质量、安全、风格一致性、是否符合党委方案）。第二步：运行项目的测试套件、linter、type checker。不通过则出具驳回报告，列明具体问题和修改要求。"
- **输入**: 所有代码变更 + 党委方案（对照验证）
- **输出**: `{ verdict: "通过|驳回", review: "审查意见", test_results: "测试结果", issues: [...] }`

## 数据流（文件产物）

所有中间产物写入 `.tmp/swcc/`：

```
.tmp/swcc/
├── zhongban-report.md        # 中办分拣报告
├── zuopai-proposal.md        # 左派方案
├── youpai-proposal.md        # 右派方案
├── zhongjian-proposal.md     # 中间路线方案（大任务时）
├── dangwei-decision.md       # 党委决策方案
├── guowuyuan-tasks.json      # 国务院子任务清单
├── buwei-{N}-result.md       # 各部委执行结果
└── jiwei-verdict.md          # 纪委审查报告
```

## 关键设计决策

1. **主 agent 仅调度** — 不扮演任何实质角色，只负责串联流程和传递数据
2. **直接管道** — 任务→代码，中间产物存本地 .tmp/
3. **动态协商深度** — 小任务 0 agent 协商、中任务 2 agent、大任务 3 agent
4. **用户可覆盖规模** — `--scale` 参数覆盖中办判定
5. **纪委驳回循环** — 不通过则重新执行，最多 2 轮（防止无限循环）
6. **举国体制** — 快速通道，跳过协商，全并行，纪委仅跑自动化
7. **所有 agent 使用 Claude Code 内置 Agent tool** — 轻量，无需外部依赖
